Jira Planning Tool — user guide

This browser extension lines up team capacity across time columns with a backlog of issues. Each issue can target a plan period so demand rolls into Planned on the capacity side. Everything is stored locally in your browser profile (Chrome / Edge).

Opening the app

From the extension toolbar menu, open Open Jira Planning Tool (wording may match the store listing). A dedicated tab hosts the UI. Import calls your Jira instance; if requests fail from the extension tab while you are logged in on Jira, the packaged content script can relay search traffic from a normal Jira tab on the same origin.

Main layout

  • Capacity — people, roles, working time, and derived metrics per period column.
  • Backlog for planning — issues, estimates, period assignment, and (depending on grouping) how demand maps to roles or members.
  • Toolbar: Create Plan, the plan dropdown (after at least one plan exists), a topbar Export icon (JSON / XLSX for the whole plan), and ⚙️ for Settings. A text link to Instructions is in the footer.

Plans

  • Each plan keeps its own periods, capacity rows, backlog, and settings (estimation type, grouping, Jira defaults, optional buffers, and so on).
  • When there is no plan yet, or capacity has no period columns, a blurred overlay appears on Capacity with a primary action (Create Plan or Create period in the rare “plan without periods” case).
  • The Backlog tab shows a similar overlay until the plan has at least one backlog row; then the backlog toolbar appears above the table (import, filter, density, bulk period actions, selection count, delete). Export is handled from topbar.

Creating a plan

Create Plan opens a dialog: plan name, Start period (quarter + year for the first anchor column), Estimation type (Story Points or Man-days), Resource grouping type (By team, By roles, or By member), and optional sprint/buffer setup.

Story Points options at creation

  • Team Story Points per day — Use Team average (mean of member inputs later) or Use fixed Team value (one number for the team cell). Fixed mode requires a numeric team value before Create.

Use Sprints Planning

  • Checking Use Sprints Planning enables Sprints settings. There you add sprint rows, set Working days per sprint, and Apply. On Create, sprint period columns are generated for the start quarter (same anchor quarter / year as your start period), each with those working days. The anchor quarter is replaced by sprint periods (no extra quarter artifact in new sprint plans).
  • While this box is checked, Man-days is disabled (sprint-style columns are intended for Story Points), and Working days for all rows in the dialog is disabled — per-sprint working days replace that default for sprint columns. Working days accepts multi-digit values and validates positive whole numbers on Apply.

Use Buffers

  • Checking Use Buffers enables Buffer settings. Add buffer lines (name + %). The All Buffers row shows the total percentage stored on the plan.
  • When buffers are on, capacity shows Available balance with buffers: the app scales the remaining balance after planned work by (1 − totalBuffer% / 100), so a higher total reduces the usable slack.

Sprint and buffer dialogs are part of plan creation (configure before clicking Create). To change them later you would need a new plan or a future settings flow if one is added.

Period columns

  • + (floating control on the Capacity side) appends the next calendar quarter in sequence after the rightmost quarter column (Q1→Q2→…→Q4→Q1 next year).
  • If the app ever opens with a plan that has zero periods, the overlay’s Create period opens a dialog to pick quarter and year manually.
  • Remove a period column with the delete control in that period’s header on the capacity table (confirmation may apply). Deleting a period removes its column from capacity and the period list used in backlog dropdowns.

Capacity tab

Rows are team members (name, role, load %, per-period working inputs, estimation per day, and roll-ups). Use + beside the table to add a member row. The Role column includes + Add role… to append to the plan’s role catalog via a small dialog.

  • Toolbar controls: member/role quick filter, Delete selected, and View. First column checkboxes select rows (header checkbox selects all).
  • Bottom stats strip shows Total rows, Filtered rows, and Selected rows.
  • View — Full shows every column, including days off and per-member detail where the layout defines them. Compact hides days off and per-member cells so the header stays aligned with fewer cells.
  • The table keeps natural column width; if it is wider than the panel, scroll horizontally inside the table area. For long tables, vertical scroll is also inside the white table area. Extra bottom space is reserved so the floating + is not blocked.
  • By team — merged team metrics where applicable; backlog demand for an issue is spread evenly across all member rows when a period is selected.
  • By roles — capacity is organized by role totals; backlog rows carry per-role split % and story points. In Full view, Planned and Available balance columns show role totals only (not per-member sub-columns), while available capacity and story points per day still expose member-level inputs.
  • By member — same capacity grid style as by team but labels emphasize Per member; backlog uses Member by roles so each role’s share of an issue can be sent to a chosen member row.
  • Team-level Story Points per day (when Story Points + by team) is edited in the merged team cell; changing it switches that period to manual team mode. For By member, edit each member’s story points per day separately.
  • If the plan includes sprint period columns, Working days for all rows in Settings is disabled — sprint cells already carry their own working days.

Backlog tab

  • Toolbar (when the table is visible): Import backlog from Jira; a Filter box (matches issue key or summary in the browser only); Period for selected rows (applies immediately on selection change); Delete selected; and View (Full / Compact, saved per plan). If any row has invalid role splits in By roles or By member, a yellow notice appears under the toolbar.
  • Bottom stats strip shows Total items, Filtered items, and Selected items.
  • The checkbox and Key columns stay pinned on the left when you scroll the table horizontally.
  • Columns include key, summary, status, priority, type, estimate, and Period (which plan period receives the issue’s demand). Empty period means the issue does not add to Planned on capacity.
  • Summary uses a fixed width so long titles wrap; scroll horizontally when there are many columns. For long lists, vertical scroll is also inside the white table area.
  • By roles — per role: Split (%) and Story Point. You may edit either side: changing % recomputes role story points from the row total; changing role story points recomputes that role’s %. Invalid combinations (splits > 100% or role points above the row total) highlight in red; hover for a short English tooltip.
  • By member — for each role, set split % and pick the capacity row (Member) that receives that share.
  • First-column checkboxes select rows (header selects all). Delete selected removes the chosen issues after confirmation (the dialog states how many issues are affected, and shows the key when only one row is selected).
  • Re-import merges on normalized keys (e.g. PROJECT-123): existing keys refresh Jira-sourced fields; new keys append. Locally deleted issues reappear if your JQL returns them again.

Settings (gear)

  • Estimation type — Story Points vs Man-days for metrics and editable backlog units. Changing it can hide imported values that were stored with a different semantic (estimation kind); re-import or re-enter estimates if needed.
  • Story Points — Team Story Points per day (team average vs fixed team value + numeric team value when fixed).
  • Working days for all rows — default working days for capacity rows and new rows (disabled when sprint columns exist).
  • Load (%) for all rows — default allocation; Save applies the chosen percentage to every capacity row (per-row overrides are not preserved across Save).
  • Resource grouping type — same three modes as at plan creation; affects both Capacity and Backlog rendering and how planned demand is attributed.
  • Default % SP by roles (visible for Story Points + By roles or By member) — one percentage per catalog role; totals must equal 100%. Editing the first role’s field redistributes the remainder evenly across the others (last row absorbs rounding). Empty backlog split cells pick up these defaults when you save Settings, after import, or when splits are first created.
  • Roles for this plan — rename, remove, or use Add role to extend the catalog. Removing a role clears it from capacity rows that referenced it. Changes apply when you click Save.

A modal titled Set estimation per day for all rows exists for bulk edits in Story Points mode; some toolbar shortcuts instead jump you to Settings to adjust the team story-points-per-day mode — follow the on-screen message.

Jira import

Open Import backlog from Jira and provide:

  • Jira Base URL — the same origin you use in the browser (saved on the plan for the next import).
  • Input JQL request — identical idea to the Issue Navigator (last JQL is remembered per plan).
  • Jira estimation field type — choose whether the field id below is Story Points (numeric) or Man-days (time / original estimate). This choice is independent of the plan’s Settings estimation type: rows remember what they were imported as so the UI does not mix units. The hint in the dialog explains that mismatched rows stay hidden until you re-import or fix the estimate.
  • Field id text box — e.g. customfield_10016 for many Story Points fields; Man-days may default to timeoriginalestimate when left empty (see inline hints).

Progress and issue counts display in the dialog. Merge behavior is keyed on normalized issue keys, not a full replace of the backlog. After a successful import, the status message may add short Data notes (for example, how many issues had no estimate in Jira).

Where data lives

Plans persist in chrome.storage.local. Clearing extension data or uninstalling can remove plans — keep separate copies of anything critical. Velocity fields exist in the data model for future use and are not surfaced as a separate workflow yet.

Jira Planning Tool — инструкция пользователя

Расширение для браузера сводит ёмкость команды по колонкам времени с бэклогом задач. У каждой задачи можно выбрать период плана, чтобы спрос учитывался в колонке Planned на стороне ёмкости. Все данные хранятся локально в профиле браузера (Chrome / Edge).

Как открыть приложение

Через меню расширения на панели инструментов выберите действие вроде Open Jira Planning Tool (точная формулировка может совпадать с описанием в каталоге). Интерфейс открывается в отдельной вкладке. Импорт обращается к вашему Jira; если запрос из вкладки расширения не проходит при активной сессии в Jira, в комплекте может использоваться сценарий с вкладкой Jira того же origin (контент-скрипт).

Основная структура экрана

  • Capacity — люди, роли, рабочее время и производные показатели по колонкам периодов.
  • Backlog for planning — задачи, оценки, выбор периода и (в зависимости от режима группировки) распределение спроса по ролям или участникам.
  • В шапке: Create Plan, выпадающий список планов (после появления хотя бы одного плана), иконка Export (JSON / XLSX для всего плана), ⚙️ — Settings. Текстовая ссылка Instructions находится в футере.

Планы

  • У каждого плана свои периоды, строки ёмкости, бэклог и настройки (тип оценки, группировка, последние параметры импорта Jira, опциональные буферы и т.д.).
  • Если плана ещё нет или у выбранного плана нет колонок периодов на Capacity, таблица закрыта оверлеем с основным действием (Create Plan или редкий случай Create period, когда план есть, но периодов ноль).
  • На вкладке бэклога оверлей показывается, пока в плане нет ни одной строки бэклога; после появления строк над таблицей показывается панель инструментов бэклога (импорт, фильтр, плотность, массовый период, счётчик выбора, удаление). Экспорт вынесен в верхнюю панель.

Создание плана

По кнопке Create Plan открывается диалог: имя плана, Start period (квартал и год первой опорной колонки), Estimation type (Story Points или Man-days), Resource grouping type (By team, By roles, By member), а также опционально спринты и буферы.

Story Points при создании

  • Team Story Points per day — Use Team average (позже по значениям участников) или Use fixed Team value (одно число для ячейки команды). Во втором случае перед Create нужно ввести числовое значение команды.

Use Sprints Planning

  • Флажок Use Sprints Planning включает кнопку Sprints settings. В таблице задаются строки спринтов и Working days на спринт, затем Apply. По нажатию Create для стартового квартала (тот же квартал/год, что в Start period) создаются колонки спринтов с этими рабочими днями. Опорный квартал заменяется спринт-периодами (лишний квартальный артефакт для нового sprint-плана не остаётся).
  • Пока флажок включён, вариант Man-days недоступен, а поле Working days for all rows в диалоге отключено — для спринтов рабочие дни задаются по строкам в настройках спринтов, а не одним общим числом в форме создания. Поле Working days поддерживает многозначный ввод и валидируется как положительное целое число при Apply.

Use Buffers

  • Флажок Use Buffers включает Buffer settings. Добавляются строки буфера (название + %). Строка All Buffers показывает суммарный процент, который сохраняется в плане.
  • При включённых буферах в таблице ёмкости подпись Available balance with buffers: к остатку после плана применяется множитель (1 − суммарный% / 100), то есть больший суммарный % сильнее уменьшает запас баланса.

Диалоги спринтов и буферов относятся к этапу создания плана (настройте до нажатия Create). Изменить те же параметры позже можно только создав новый план или дождавшись отдельного сценария в настройках, если он появится в версии.

Колонки периодов

  • Кнопка + у области Capacity добавляет следующий календарный квартал после самой правой квартальной колонки (Q1→…→Q4→Q1 следующего года).
  • Если план открылся без периодов, оверлей с Create period открывает диалог ручного выбора квартала и года.
  • Удалить колонку периода можно элементом удаления в заголовке этой колонки в таблице Capacity (при необходимости с подтверждением). Вместе с колонкой период исчезает из выпадающих списков в бэклоге.

Вкладка Capacity

Строки — участники (имя, роль, нагрузка %, рабочие дни по периодам, оценка в день, агрегаты). Кнопка + у таблицы добавляет строку. В колонке роли есть пункт + Add role… для добавления роли в каталог через небольшой диалог.

  • Элементы панели: быстрый фильтр по member/role, Delete selected и View. Первая колонка — чекбоксы выбора строк (в шапке — выбрать все).
  • Нижняя серая панель показывает Total rows, Filtered rows и Selected rows.
  • Поле View — Full показывает все колонки, включая выходные и детализацию «на человека», где вёрстка это предусматривает. Compact скрывает выходные и ячейки per member, чтобы заголовок совпадал с числом ячеек в строках.
  • Ширина таблицы по содержимому; при переполнении панели — горизонтальная прокрутка внутри области таблицы. При большом числе строк вертикальная прокрутка тоже остаётся внутри белой области. Снизу оставлен запас, чтобы плавающая кнопка + не перекрывала строки.
  • By team — где уместно, блоки команды объединены; спрос по задаче с выбранным периодом делится поровну между всеми строками ёмкости.
  • By roles — ёмкость сгруппирована по ролям; в бэклоге задаются доли и story points по ролям. В режиме Full колонки Planned и Available balance показывают только итоги по ролям (без подколонок на каждого человека), при этом доступная ёмкость и story points в день по-прежнему ведутся на уровне участников.
  • By member — сетка ёмкости как у команды, но с акцентом на Per member; в бэклоге блок Member by roles: для каждой роли — доля и выбор строки участника, куда относится эта доля.
  • Ячейка Story Points per day на уровне команды (Story Points + by team) редактируется в объединённой ячейке; изменение переводит период в режим ручного значения для команды. В режиме By member значения в день задаются по каждому участнику.
  • Если в плане есть колонки спринтов, поле Working days for all rows в Settings отключено — рабочие дни уже заданы в ячейках спринтов.

Вкладка Backlog for planning

  • Панель над таблицей (когда таблица видна): импорт из Jira; поле фильтра (поиск по ключу или summary только в браузере); Period для выбранных строк (применяется сразу при выборе значения); Delete selected; View (Full / Compact, сохраняется в плане). При ошибках долей в режимах By roles / By member под панелью показывается жёлтое предупреждение.
  • Нижняя серая панель показывает Total items, Filtered items и Selected items.
  • Колонки чекбокс и Key закреплены слева при горизонтальной прокрутке таблицы.
  • Колонки: ключ, summary, статус, приоритет, тип, оценка, Period — в какой период плана относится спрос. Без периода задача не увеличивает Planned на Capacity.
  • У Summary фиксированная ширина — длинный текст переносится внутри колонки; при большом числе колонок используйте горизонтальную прокрутку. При большом числе строк вертикальная прокрутка тоже остаётся внутри белой области таблицы.
  • By roles — для каждой роли: Split (%) и Story Point. Можно править и процент, и SP: изменение % пересчитывает ролевые SP от общей оценки строки; изменение SP роли пересчитывает её %. Некорректные комбинации (сумма долей > 100% или сумма ролевых SP больше оценки строки) подсвечиваются красным; подсказка при наведении на английском.
  • By member — для каждой роли: доля (%) и выбор участника (Member), на чью строку ёмкости относится эта доля.
  • Первая колонка — чекбоксы (в шапке — выбрать все). Delete selected удаляет выбранные строки после подтверждения (в диалоге указано число задач; для одной строки показывается ключ).
  • Повторный импорт объединяет по нормализованному ключу (например, PROJECT-123): существующие ключи обновляются, новые добавляются. Локально удалённая задача появится снова, если снова попадёт в выборку JQL.

Настройки (шестерёнка)

  • Estimation type — Story Points или Man-days для метрик и редактируемых оценок в бэклоге. Смена типа может скрыть импортированные значения другой семантики; тогда помогут повторный импорт или ручной ввод.
  • Для Story Points — Team Story Points per day (среднее по команде или фиксированное значение + число при фиксированном режиме).
  • Working days for all rows — базовые рабочие дни для строк и новых строк (отключается, если в плане есть спринтовые колонки).
  • Load (%) for all rows — загрузка по умолчанию; по кнопке Save выбранный процент применяется ко всем строкам ёмкости (индивидуальные отличия при сохранении не сохраняются).
  • Resource grouping type — те же три режима, что при создании плана; влияет на обе вкладки и на то, как спрос попадает в Planned.
  • Default % SP by roles (видно при Story Points и группировке By roles или By member) — процент по каждой роли из каталога; сумма должна быть 100%. Редактирование первой строки перераспределяет остаток поровну по остальным ролям (последняя строка забирает округление). Пустые ячейки Split (%) в бэклоге заполняются этими значениями при сохранении настроек, после импорта и при первом появлении долей.
  • Roles for this plan — переименование, удаление, Add role. Удаление роли сбрасывает её у строк, где она была выбрана. Изменения вступают в силу после Save.

В приложении есть модальное окно Set estimation per day for all rows для массовых правок в режиме Story Points; отдельные действия в интерфейсе могут вместо этого открыть Settings с подсказкой настроить режим story points в день для команды — ориентируйтесь на сообщение в статусной строке.

Импорт из Jira

По кнопке Import backlog from Jira укажите:

  • Jira Base URL — тот же адрес origin, что в браузере (сохраняется в плане для следующего открытия диалога).
  • Input JQL request — как в навигаторе задач (последний JQL запоминается в плане).
  • Jira estimation field type — отдельно от типа оценки в Settings: это что именно означает поле в Jira (числовые Story Points или Man-days по времени). У импортированных строк запоминается вид оценки, чтобы не смешивать единицы при смене типа плана; подсказка в диалоге описывает скрытие несовпадающих строк до повторного импорта или правки.
  • Поле идентификатора — например customfield_10016 для SP; для Man-days пустое значение может подставлять стандартное поле времени (см. подсказки в форме).

В диалоге отображаются прогресс и число задач. Импорт не затирает весь бэклог, а сливает по ключу. После успешного импорта в статусной строке могут появиться краткие заметки по данным (например, сколько задач без оценки в Jira).

Где хранятся данные

Планы лежат в chrome.storage.local. Очистка данных расширения или удаление расширения может стереть планы — заранее сохраняйте важное вне инструмента. Поля velocity в модели данных зарезервированы под будущие сценарии и пока не вынесены отдельным сценарием в интерфейсе.

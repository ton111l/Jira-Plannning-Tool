# BRD + SRS/FRD v1 — Jira Planning Extension (MVP)

## 1) Назначение
Расширение для браузера упрощает квартальное планирование: расчет capacity команды и формирование backlog для планирования.

## 2) Цель MVP
- Дать возможность указывать команду, роли и параметры загрузки.
- Дать возможность планировать период в формате квартала.
- Дать возможность формировать backlog вручную и через импорт из Jira.

## 3) Пользователи
- Delivery Manager
- PM
- Team Lead
- Scrum Master

Фокус оценивания backlog в MVP: Story Points.

## 4) Scope MVP
- Только кварталы: `Qx + Year`.
- Одна команда в одном плане.
- Вкладки: `Capacity` и `Backlog for planning`.
- Управление планами: `Create Plan` и `Select Plan`.
- Локальное хранение планов и восстановление последнего выбранного плана.
- Импорт задач из Jira по JQL + ручное редактирование.

## 5) Out of Scope
- Месяцы и спринты.
- Экспорт/шаринг (CSV/JSON).
- Синхронизация между устройствами.
- Несколько Jira-инстансов.
- Конверсия оценок (часы/дни/SP).
- Автоподгрузка праздничных календарей.

## 6) Функциональные требования

### FR-01. Запуск
По клику на иконку расширения открывается отдельная вкладка приложения (не popup).

### FR-02. Планы
- Создание плана (`Create Plan`) с именем и периодом.
- Выбор плана (`Select Plan`).
- Редактирование существующих планов.
- Хранение локально.

### FR-03. Capacity
Поля для участника:
- Name
- Role
- Load (%)
- Days off
- Working days
- Planned capacity

Формула:
`Planned capacity = (Working days - Days off) * Load%`

### FR-04. Динамика таблицы Capacity
- Добавление строк снизу: добавляет сотрудника для уже добавленных кварталов.
- Добавление кварталов справа: добавляет кварталы без добавления новых строк сотрудников.

### FR-05. Период
- Формат периода: `Q1..Q4` + `Year`.
- Working days вводятся вручную.
- Праздники автоматически не подтягиваются.

### FR-06. Backlog
Обязательные поля:
- Key
- Summary
- IssueType
- Priority
- Estimation

Возможности:
- Ручное редактирование ячеек.
- Добавление строк снизу.
- Импорт из Jira по JQL.
- Редактирование импортированных данных вручную.

### FR-07. Jira Data Server Import
- Один Jira instance.
- Использование активной браузерной сессии пользователя.
- Импорт по `JQL + Import`.
- Обработка ошибок 401/403/сетевых с fallback-подсказкой.

## 7) Нефункциональные требования

### NFR-01. Закрытый контур (критично)
- Только нативный JS/CSS или локально встроенные библиотеки.
- Нельзя грузить скрипты, стили, шрифты и библиотеки из интернета (CDN и внешние URL запрещены).

### NFR-02. Производительность
- До 50 участников без заметных лагов.
- До 200 задач в backlog без заметных лагов.

### NFR-03. Надежность
- Данные сохраняются при перезапуске браузера.
- Последний выбранный план восстанавливается автоматически.

### NFR-04. Безопасность
- Не логировать cookie, session ID и чувствительные данные Jira.
- Хранить только необходимые поля задач для планирования.

## 8) Модель данных (минимум)
- `Plan`: id, name, teamName, periods, capacityRows, backlogRows, updatedAt
- `Period`: quarter, year
- `CapacityRow`: memberName, role, loadPercent, daysOffByPeriod, workingDaysByPeriod, plannedCapacityByPeriod
- `BacklogRow`: key, summary, issueType, priority, estimationSP, source
- `AppState`: lastSelectedPlanId

## 9) Acceptance Criteria
1. Можно создать, выбрать и редактировать план.
2. Capacity поддерживает добавление сотрудников вниз.
3. Capacity поддерживает добавление кварталов справа без роста строк.
4. Формула capacity корректно пересчитывается.
5. Backlog можно редактировать вручную.
6. Backlog можно импортировать из Jira по JQL.
7. Импортированные строки можно редактировать вручную.
8. Ошибки Jira (401/403/сеть) обрабатываются понятным сообщением.
9. Последний выбранный план восстанавливается после повторного открытия.
10. Работоспособность сохраняется на 50 участниках и 200 задачах.

## 10) Краткий план реализации
1. Каркас extension и отдельная вкладка приложения.
2. Модуль локального хранения планов.
3. UI вкладки `Capacity` и логика формулы.
4. UI вкладки `Backlog for planning`.
5. Jira import по JQL и обработка ошибок.
6. Полировка UX и проверки на целевых объемах.

## 11) UI описание (MVP)

### 11.1 Общая структура страницы
- Верхняя панель: логотип/название `Jira Planning Tool`, кнопки `Create Plan`, `Select Plan`, иконка настроек.
- Основная рабочая область с вкладками:
  - `Capacity`
  - `Backlog for planning`
- Все экраны доступны в отдельной вкладке браузера, открываемой расширением.

### 11.2 Модальное окно Create Plan
Поля:
- `Name` (название плана)
- `Start period`: выбор квартала (`Q1..Q4`) и года (`YYYY`)

Действия:
- `Create` — создает план и делает его активным.
- `Cancel/Close` — закрывает окно без изменений.

Правила:
- Имя плана обязательно.
- Период обязателен и хранится в формате `Qx + Year`.

### 11.3 Селектор Select Plan
- Выпадающий список/меню существующих планов.
- При выборе плана:
  - загружаются все сохраненные данные `Capacity` и `Backlog`;
  - выбранный план становится `lastSelectedPlan`.

### 11.4 Вкладка Capacity
Таблица состоит из двух логических зон:
- Левый блок (постоянные поля сотрудника):
  - `#` (индекс)
  - `Member`/`Name`
  - `Role`
  - `Load (%)`
- Правый блок (повторяется по каждому кварталу):
  - `Days off`
  - `Working days`
  - `Planned capacity`

Кнопки/действия:
- `+ Row` (снизу): добавить сотрудника.
- `+ Quarter` (справа): добавить блок колонок нового квартала.
- Редактирование ячеек по клику.

Поведение:
- При изменении `Load`, `Days off`, `Working days` сразу пересчитывается `Planned capacity`.
- Добавление квартала не добавляет автоматически новые строки сотрудников.
- Скролл таблицы горизонтальный при большом числе кварталов.

### 11.5 Вкладка Backlog for planning
Таблица backlog:
- `#`
- `Key`
- `Summary`
- `IssueType`
- `Priority`
- `Estimation`

Кнопки/действия:
- `Import` — открывает окно импорта из Jira.
- `+ Row` — добавить пустую строку вручную.
- Inline-редактирование любой ячейки.

### 11.6 Модальное окно Import candidate's backlog
Элементы:
- Поле/textarea `Input JQL request`
- Счетчик `Issues count`
- Кнопка запуска импорта `Import`
- Индикатор прогресса (progress bar)

Сценарий:
1. Пользователь вводит JQL.
2. Нажимает `Import`.
3. Показывается прогресс и число найденных задач.
4. По завершении данные добавляются/обновляются в таблице backlog.

Ошибки:
- `401/403`: сообщение о необходимости авторизации в Jira.
- Сетевые ошибки: предложение повторить запрос.
- Пустой результат: корректное состояние `0 issues`.

### 11.7 Состояния UI
Обязательные состояния:
- `Empty` (нет данных)
- `Loading` (идет импорт/обработка)
- `Success` (операция завершена)
- `Error` (ошибка операции)

Принципы:
- Сообщения понятные и короткие.
- Действия восстановления доступны рядом с сообщением (`Retry`, `Close`, `Re-login hint`).

### 11.8 UX ограничения MVP
- Нет drag-and-drop.
- Нет сложной фильтрации и сортировки.
- Нет экспорта файлов.
- Нет многоинстансовой Jira-конфигурации.

### 11.9 UI критерии приемки
1. Пользователь может создать план через модальное окно.
2. Пользователь может переключаться между планами через `Select Plan`.
3. Во вкладке `Capacity` доступны добавление строк и кварталов.
4. Во вкладке `Backlog` доступны ручной ввод и импорт.
5. Ошибки импорта отображаются без потери уже введенных данных.

## 12) Подход к разработке и сборке

### 12.1 Этап отладки (до сборки расширения)
- На начальном этапе вся логика и UI отлаживаются без финальной сборки extension-пакета.
- Цель этапа: быстро проверять поведение экранов, расчеты и интеграцию с Jira API без усложнения пайплайна сборки.
- На этом этапе допускается запуск в режиме разработки и ручная проверка сценариев.

### 12.2 Этап сборки (после готовности версии)
- Переход к сборке расширения выполняется только после стабилизации функционала MVP.
- Критерий перехода: реализованы и проверены ключевые сценарии, пройдены acceptance criteria.
- После перехода выполняется упаковка в готовую версию расширения для установки пользователем.
